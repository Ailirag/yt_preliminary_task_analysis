# ИИ-анализатор задач в Yandex Tracker (очередь ONE «1С Разработка»)

## Коротко

ИИ-анализатор — это сервис, который автоматически проводит **предварительный анализ ошибок 1С**:
читает задачу (описание, комментарии, скриншоты, связанные задачи, документацию в вики), сам ходит
по **коду конфигурации 1С** нужной системы и формирует отчёт — что за проблема, где вероятная причина
в коде, насколько сложно исправить и черновик решения.

Результат прикладывается к задаче **отдельной подзадачей** и служит стартовой точкой для разработчика.
Это **не окончательный вердикт**, а помощник для ускорения первичной диагностики.

> ⚠️ Отчёт ИИ носит рекомендательный характер и требует проверки разработчиком.

---

## Для разработчика: как отправить задачу на анализ

### Шаг 1. Повесьте на баг тег

Добавьте задаче типа **«Ошибка»** тег:

```
к_анализу_ии
```

Ничего больше делать не нужно. Тег — единственный триггер.

### Что произойдёт и когда

- В течение **~30 секунд** демон-анализатор заметит задачу и возьмёт её в работу.
- Разбор одной задачи занимает обычно **несколько минут** (зависит от сложности и объёма связанных материалов).
- Пока идёт разбор, задача не блокируется — можно работать с ней как обычно.

### Где смотреть результат

- У задачи появится **подзадача** с названием `[ИИ анализ] ONE-XXXX: <тема>` и компонентой **«ИИ анализ»**
  (тип «Анализ информации»). Внутри — полный отчёт.
- На самой задаче:
  - снимется тег `к_анализу_ии`;
  - появится тег **`ИИ_анализ_проведен`**;
  - появится оценка сложности: **`ИИ-простая`** или **`ИИ-сложная`**.

Если анализатор **не смог определить, к какой системе 1С относится задача**, вместо разбора он оставит
**комментарий** «Анализ невозможен: невозможно установить целевую систему анализа…» и повесит тег
**`ИИ_анализ_невозможен`** (см. раздел «Определение системы 1С»).

## Что содержит ИИ-отчёт

Отчёт в подзадаче структурирован:

| Раздел | Что внутри |
|---|---|
| **Вердикт доверия** | Оценка обоснованности разбора 0–100 и уровень: *доверять / частично / не доверять*. Считается автоматически по тому, насколько анализ опирался на реальный код, а не на догадки. |
| **Резюме** | Что происходит, где и почему (если установлено). |
| **Воспроизведение** | Шаги/условия воспроизведения или отметка, что не установлены. |
| **Вероятные причины** | Гипотезы с уровнем уверенности и обоснованием (на чём основаны). |
| **Затронутые объекты конфигурации** | Таблица: объект метаданных → модуль → процедура → роль в проблеме. |
| **Черновик решения / шаги диагностики** | Для простых — конкретные шаги правки; для сложных — план диагностики. |
| **Чего не хватает в постановке** | Вопросы к постановщику. |
| **Ссылки на код** | Что именно смотрели в коде. |
| **Метрики** | Модели, число шагов по коду, токены, ориентировочная стоимость. |

## Когда задача берётся в анализ (полный список условий)

Демон опрашивает очередь **ONE** каждые 30 секунд и берёт задачу, только если выполнено **всё**:

| Условие | Требование |
|---|---|
| Тип задачи | **Ошибка** |
| Resolution | пусто (задача не закрыта) |
| Тег `к_анализу_ии` | **есть** |
| Тег `ИИ_анализ_проведен` | отсутствует |
| Тег `ИИ_анализ_невозможен` | отсутствует |
| ИИ-подзадача | ещё не создавалась (иначе повторный анализ не запускается) |

Если у задачи уже есть ИИ-подзадача — повторный разбор не делается (только «долечиваются» теги).
Чтобы **переанализировать** — используется отдельный ручной запуск с флагом принудительного пересоздания
(см. раздел администратора).

## Определение системы 1С (мульти-система)

Анализатор работает с **несколькими системами 1С** и перед разбором сам определяет, к какой из них
относится задача, чтобы искать причину в правильном коде.

**Поддерживаемые системы:**

| Система | Как обычно распознаётся (компонента / текст) |
|---|---|
| **Управление Торговлей (УТ)** | компоненты `1С УТ`, `УТ`; слова «УТ», «управление торговлей», «розница» |
| **Бухгалтерия предприятия (БП)** | компонента `1С БП`; слова «БП», «бухгалтерия» |
| **Документооборот (ДО)** | компоненты `1С ДО`, `Документооборот`; слово «документооборот» |
| **Зарплата и управление персоналом (ЗУП)** | компонента `1С ЗУП`; слова «ЗУП», «зарплата», «кадры» |
| **Управление мастер-данными (MDM/НСИ)** | слова «MDM», «НСИ», «мастер-данные», «pimcore» |

### Как помочь анализатору определить систему

- Проще всего — **добавить задаче соответствующую компоненту** (`1С УТ`, `1С БП`, `1С ДО`, `1С ЗУП`).
- Либо явно **упомянуть систему в описании** (например, «Ошибка в УТ при проведении…»).

### Если систему определить нельзя

Анализатор оставит комментарий *«Анализ невозможен: невозможно установить целевую систему анализа.
Укажите систему 1С в описании задачи или добавьте компоненту, указывающую на систему.»* и повесит тег
**`ИИ_анализ_невозможен`**. После того как вы уточните систему (компонента/описание) и **снимете этот тег**
(или заново поставите `к_анализу_ии`), задачу можно отправить на анализ повторно.

### Интеграционные задачи

Если задача затрагивает **несколько систем** (например, обмен УТ↔БП), анализатор определит все
подходящие системы и будет анализировать код каждой из них.

## Как это работает под капотом

1. **Минизапуск** — короткий и дешёвый запрос к LLM: по теме, описанию и компонентам определяется
   целевая система(ы) 1С.
2. **Сбор досье** — описание, комментарии, связи, вложения. **Скриншоты** распознаются vision-моделью
   и превращаются в текст. Подхватывается страница вики из поля «Ссылка на документацию».
3. **Агентный разбор кода** — модель-аналитик сама ходит по коду выбранной системы (полнотекстовый и
   точный поиск, структура объектов, граф вызовов, чтение модулей) и по связанным задачам/вики.
   На разбор отводится ограниченный бюджет обращений.
4. **Отчёт и вердикт** — формируется markdown-отчёт, считается вердикт доверия и метрики стоимости.

Используемые модели (профиль по умолчанию — Yandex): аналитик `deepseek-v4-flash`, распознавание
скриншотов `qwen3.6-35b-a3b`. Профиль можно переключать (есть также профиль на моделях z.ai).

---

## Системы 1С и актуальность кода

- Код каждой системы — это **зеркало соответствующего git-репозитория** конфигурации, которое сервис
  клонирует и держит у себя.
- Зеркала **обновляются по расписанию** (в окна вне пиков анализа), чтобы анализ шёл по актуальному коду,
  но версия не менялась «под ногами» посреди разбора.
- Поисковый индекс по коду дообновляется автоматически.

---

## Анализ функциональных требований (ФТ)

Кроме ошибок, анализатор умеет разбирать **готовые ФТ** (отдельный режим):
- триггер — тег **`К анализу ИИ`**;
- **обязательно** заполненное поле «Ссылка на документацию» (иначе задача пропускается);
- результат — подзадача `[ИИ анализ ФТ] …` с оценкой полноты требования, проекцией на конфигурацию и
  черновиком плана реализации; тег готовности `ИИ_анализ_ФТ_проведен`.

> В штатном фоновом режиме демон настроен на **ошибки**. Разбор ФТ запускается отдельно (см. администратора).

---

## Ограничения и дисклеймер

- Это **предварительный** автоматический анализ, возможны неточности и ошибочные гипотезы.
- Ориентируйтесь на **вердикт доверия**: низкий балл означает, что анализ слабо опирался на код.
- Финальное решение и его корректность — за разработчиком.
- Анализатор **только читает**: он не изменяет код и не меняет поля задач (кроме своих тегов, комментария
  о невозможности анализа и создания собственной подзадачи).

---

## Безопасность и приватность

- **Write-guard:** на чужих задачах разрешены только теги и комментарий; полный доступ (тело, поля) — только
  к собственной ИИ-подзадаче, созданной в текущем прогоне. У модели write-инструментов нет — её результат
  только JSON-отчёт.
- Содержимое задач, комментариев, вики и скриншотов трактуется как **данные для анализа, а не инструкции**
  (защита от инъекций).
- **Токены и ключи** хранятся только в переменных окружения и никогда не попадают в логи/отчёты.
- Данные компании (выгрузки кода, журналы прогонов) не попадают в публичные репозитории.
- **Двойной предохранитель записи:** реальная запись в трекер идёт только в боевом режиме (`live`) и при
  явном флаге запуска. В режиме `dry-run` результат пишется только в локальный журнал.

## Для администратора: эксплуатация

### Где развёрнуто

- Хост: прод-сервер (Linux), запуск через **Docker Compose** (сервисы `tracker-analyzer` —
  демон, и `onec-sync` — обновление зеркал кода).
- Каталог проекта: `/home/adm_garipov_ir/analyzer`; данные и индексы — на `/mnt/storage/analyzer/…`.
- Секреты (токены трекера, ключи LLM, креды git к corp-репозиториям) — в env-файле, не в git.

### Команды

```bash
cd /home/adm_garipov_ir/analyzer
COMPOSE="docker compose -f deploy/docker-compose.yml"

$COMPOSE exec analyzer /app/.venv/bin/analyzer status   # статус демона, режим, бюджет, глубина очереди
$COMPOSE logs -f analyzer                                # логи разбора
$COMPOSE logs -f onec-sync                               # логи обновления зеркал
$COMPOSE restart analyzer                                # мягкий перезапуск (индексы не пересобираются)
$COMPOSE up -d --build                                   # применить обновление кода/конфига
$COMPOSE down                                            # остановить (graceful, до 10 мин на текущий разбор)
```

Ручной разбор одной задачи (в обход демона; профиль по умолчанию — yandex):

```bash
$COMPOSE exec analyzer /app/.venv/bin/analyzer bugs --issue ONE-XXXX --limit 1
# повторный разбор поверх существующей ИИ-подзадачи:
$COMPOSE exec analyzer /app/.venv/bin/analyzer bugs --issue ONE-XXXX --limit 1 --force
```

### Журналы и аудит

- `journal/runs.jsonl` — все прогоны (действие, доверие, стоимость, токены).
- `journal/writes.jsonl` — аудит всех записей в трекер (в т.ч. намерения в dry-run).
- `journal/dry-run/ONE-XXXX.md` — отчёты, сформированные в dry-run.
- `work/daily_spend.json` — суточный расход.

### Тонкая настройка (`config/analyzer.yaml`)

| Параметр | Назначение |
|---|---|
| `mode` | `dry-run` (только журнал) / `live` (запись в трекер). |
| `watch.selection` | `trigger-tag` (только по тегу — безопасно) / `no-done-tag` (все незакрытые ошибки). |
| `watch.interval_s` | Период опроса, сек. |
| `watch.profile` | Профиль моделей демона (`yandex` / `z.ai` / …). |
| `watch.daily_budget` | Суточный потолок стоимости (пусто = без лимита). |
| `watch.work_hours` | Окно работы, напр. `"08:00-20:00"` (пусто = круглосуточно). |
| `bugs.trigger_authors` | Белый список авторов тега-триггера (e-mail). **Пусто = запускать может кто угодно.** |
| `systems` | Список систем 1С: имя, воркспейс, git-репозиторий, компоненты-подсказки. |

Профили и модели — в `config/providers.yaml`. Оба конфига подключены внешними файлами: правка + `restart`
применяются без пересборки образа.

### Добавить новую систему 1С

1. В `config/analyzer.yaml` → `systems` добавить блок: `name`, `workspace` (короткое имя), `repo` (git-URL),
   `components` (компоненты трекера, указывающие на систему), `aliases` (слова-подсказки для распознавания).
2. `$COMPOSE up -d --build` — при старте сервис сам сгенерирует конфигурацию, склонирует зеркало и построит
   поисковый индекс для новой системы (первый индекс строится несколько минут, далее обновляется инкрементально).

### Включить / выключить боевой режим

```bash
# включить запись в трекер:
sed -i 's/^mode: dry-run/mode: live/' config/analyzer.yaml && $COMPOSE up -d --build
# вернуть в безопасный режим:
sed -i 's/^mode: live/mode: dry-run/' config/analyzer.yaml && $COMPOSE up -d --build
```

### Ограничить, кто может запускать анализ тегом

В `config/analyzer.yaml` → `bugs.trigger_authors` указать список e-mail; тогда задача берётся, только если
тег-триггер поставил кто-то из списка. Затем `restart`.

## Частые вопросы (FAQ)

**Пометил задачу — ничего не происходит.** Проверьте: тип задачи «Ошибка»; тег ровно `к_анализу_ии`;
нет тегов `ИИ_анализ_проведен` / `ИИ_анализ_невозможен`; задача не закрыта; у неё ещё нет ИИ-подзадачи.

**Появился тег `ИИ_анализ_невозможен` и комментарий.** Анализатор не понял систему 1С. Добавьте компоненту
(`1С УТ`/`1С БП`/`1С ДО`/`1С ЗУП`) или укажите систему в описании, снимите тег `ИИ_анализ_невозможен` и
поставьте `к_анализу_ии` заново.

**Низкий вердикт доверия.** Анализ мало опирался на код (не хватило данных в постановке, неоднозначная
причина). Отчёт всё равно полезен как отправная точка; дополните постановку и при желании перезапустите.

**Нужно перепроанализировать задачу.** Администратор запускает ручной разбор с флагом `--force`.

**Разбор долгий / дорогой.** Сложные задачи с множеством связей и скриншотов разбираются дольше. Есть
суточный бюджет-лимит и окно работы (настраиваются администратором).

## Глоссарий

| Тег / компонента | Смысл |
|---|---|
| `к_анализу_ии` | Триггер: поставить, чтобы отправить ошибку на анализ. |
| `ИИ_анализ_проведен` | Анализ ошибки выполнен, создана подзадача. |
| `ИИ_анализ_невозможен` | Систему 1С определить не удалось; см. комментарий к задаче. |
| `ИИ-простая` / `ИИ-сложная` | Оценка сложности исправления. |
| `К анализу ИИ` | Триггер анализа готового ФТ (нужна ссылка на документацию). |
| `ИИ_анализ_ФТ_проведен` | Анализ ФТ выполнен. |
| Компонента «ИИ анализ» | Помечает подзадачи, созданные анализатором. |
